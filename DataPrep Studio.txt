You are Codex acting as a senior full-stack engineer and ML tooling engineer. Build a new full-stack project named DataPrep Studio.

DataPrep Studio is an expert-oriented data quality and preprocessing workbench for tabular machine learning workflows.

The core idea:
Users upload CSV datasets, inspect data quality and ML-readiness issues, build a configurable preprocessing pipeline step by step, tune transformation parameters, preview before/after effects, and export cleaned datasets plus reproducible preprocessing configs and code.

This is NOT an AutoML tool.
This is NOT a black-box “fix my data automatically” tool.
This is NOT an AI/LLM app.

It is a practical ML preprocessing workbench for users who understand data science and want control over preprocessing strategies and parameters.

The product philosophy:
- Diagnose issues clearly.
- Offer transformation options.
- Let the user choose operations and parameters.
- Preview effects before applying.
- Apply transformations in a reproducible way.
- Preserve leakage-safe train/test behavior.
- Export cleaned data, config, reports, and code.

============================================================
PROJECT GOAL
============================================================

Build an MVP called DataPrep Studio.

One-line product description:
DataPrep Studio is a configurable preprocessing workbench that helps users profile tabular datasets, diagnose data quality issues, build custom preprocessing pipelines, tune transformation parameters, preview before/after effects, and export reproducible cleaned datasets for ML modeling.

Primary users:
- Data Science interns
- Junior ML engineers
- Students working on ML projects
- Backend/data developers who want a structured data preparation workflow
- Users who understand preprocessing and want control, not automatic black-box decisions

Core user workflow:
1. Create a project.
2. Upload either:
   - one CSV dataset, or
   - a train.csv and test.csv pair.
3. Select target column and problem type.
4. Run dataset profile and ML-readiness diagnosis.
5. Inspect issues: missingness, duplicates, mixed types, outliers, high-cardinality features, target imbalance, leakage candidates, train/test drift.
6. Build a preprocessing pipeline manually:
   - drop columns
   - impute missing values
   - add missingness indicators
   - group rare categories
   - encode categorical features
   - scale numeric features
   - clip outliers
   - extract datetime features
   - clean text columns
   - remove duplicate rows
7. Tune parameters for each step.
8. Preview before/after effects.
9. Apply pipeline.
10. Export:
   - cleaned CSV files
   - preprocessing_config.json
   - preprocessing_report.md
   - generated Python preprocessing code
11. Re-run or edit pipeline later.

============================================================
ISSUE DETECTION REQUIREMENTS
============================================================

Implement services/issue_detector.py.

Function:
detect_issues(df_or_train_df, optional_test_df, column_profiles, target_column, problem_type) -> list[IssueData]

Required checks:

1. Missingness:
- critical if column missing_rate >= 0.50
- warning if missing_rate >= 0.10
- info if missing_rate > 0
Suggested actions:
- numeric imputation
- categorical imputation
- missingness indicator
- drop column if very high missingness

2. Duplicate rows:
- warning if duplicate rows exist
- critical if duplicate row rate >= 0.10
Suggested actions:
- remove duplicate rows
- inspect whether duplicates are legitimate repeated observations

3. Constant / near-constant columns:
- critical if unique_count <= 1
- warning if dominant_value_rate >= 0.95
Suggested actions:
- drop column
- leave unchanged if domain reason exists

4. High-cardinality categorical columns:
- warning if unique_count > 50 and cardinality_ratio > 0.20
- possible ID warning if cardinality_ratio > 0.90
Suggested actions:
- group rare categories
- frequency encoding
- drop ID-like column
- keep only if domain justification exists

5. Mixed type / parsing issues:
- warning if object column has many numeric-looking values but also non-numeric values
- warning if datetime parse rate is partial
Suggested actions:
- convert to numeric
- parse datetime
- replace invalid values with missing

6. Numeric outliers:
- warning if IQR outlier rate >= 0.05
- info if outlier rate > 0
Suggested actions:
- percentile clipping
- IQR clipping
- log1p transform
- leave unchanged if meaningful extreme values

7. Suspicious placeholder values:
Detect strings:
- "N/A"
- "NA"
- "unknown"
- "Unknown"
- "?"
- "-"
- "--"
- "-999"
- "null"
- "None"
Suggested actions:
- replace placeholders with missing
- then impute or leave missing

8. Target issues:
Classification:
- warning if majority_class_rate >= 0.80
- critical if minority_class_rate <= 0.02
- warning if too many target classes
Regression:
- warning for extreme outliers
- warning for high skewness
Suggested actions:
- stratified split
- macro-F1 / PR-AUC for imbalanced classification
- class weights for model training
- do not automatically resample in MVP

9. Leakage candidates:
Heuristic warnings:
- column name includes target-like terms:
  result, outcome, status, approved, accepted, label, target, score, final, after, post
- column has near-perfect correlation with target
- categorical column almost determines target
- feature name overlaps with target column name
Suggested actions:
- inspect manually
- drop from modeling if confirmed leakage
- keep only for reporting if needed

10. Train/test drift:
Only when train and test are available.
Numeric:
- mean shift
- std shift
- quantile shift
Categorical:
- unseen categories in test
- category frequency shift
- missing categories
Target:
- target distribution shift
Split:
- duplicate rows across train/test
- ID-like column overlapping across train/test
Suggested actions:
- check split strategy
- consider stratified split
- handle unknown categories
- group rare categories
- avoid leakage across entities

Each issue must include:
- severity
- category
- title
- explanation
- affected_columns
- suggested_actions

MVP operation types:

1. drop_columns
Params:

}
Behavior:

2. remove_duplicate_rows
Params:
{
  "subset_columns": [],
  "keep": "first" | "last"
}
Behavior:
- Remove duplicate rows from single/train.
- For train/test mode, apply separately to train and test.
- Warn that row removal changes target distribution.

3. replace_placeholders_with_missing
Params:
{
  "placeholders": ["N/A", "NA", "unknown", "?", "-", "-999", "null", "None"]
}
Behavior:
- Replace listed placeholder strings with NaN.

4. numeric_imputation
Params:
{
  "strategy": "mean" | "median" | "constant",
  "fill_value": 0,
  "add_indicator": true
}
Behavior:
- Fit statistic on train only in train/test mode.
- Transform train and test using train statistic.
- For single mode, fit on single dataset.

5. categorical_imputation
Params:
{
  "strategy": "most_frequent" | "constant",
  "fill_value": "__MISSING__",
  "add_indicator": false
}
Behavior:
- Fit mode on train only in train/test mode.
- Transform test with train-fitted value.

6. add_missingness_indicator
Params:
{
  "suffix": "_missing"
}
Behavior:
- Add binary indicator columns for selected columns.

7. rare_category_grouping
Params:
{
  "min_frequency": 0.01,
  "min_count": null,
  "rare_label": "__RARE__",
  "include_missing": false
}
Behavior:
- Learn frequent categories from train only.
- Replace rare/unseen categories with rare_label.
- In train/test mode, unseen test categories become rare_label.

8. one_hot_encoding
Params:
{
  "drop_first": false,
  "handle_unknown": "ignore",
  "max_categories": null
}
Behavior:
- Learn categories from train only.
- Transform train/test consistently.
- Produce predictable column names.
- If too many generated columns, warn.

9. ordinal_encoding
Params:
{
  "categories_order": {},
  "unknown_value": -1
}
Behavior:
- Use user-provided category order if given.
- Otherwise sorted order.
- Unknown test categories use unknown_value.

10. frequency_encoding
Params:
{
  "normalize": true,
  "unknown_value": 0
}
Behavior:
- Learn category frequencies from train only.
- Map categories to frequency.
- Unknown categories use unknown_value.

11. numeric_scaling
Params:
{
  "method": "standard" | "minmax" | "robust",
  "feature_range": [0, 1],
  "quantile_range": [25, 75]
}
Behavior:
- Fit scaler statistics on train only.
- Apply same stats to test.
- For MVP, pandas-based implementation is okay.
- scikit-learn scaler use is also okay.

12. outlier_clipping

PIPELINE ENGINE REQUIREMENTS
============================================================

Implement services/pipeline_engine.py.

Core concepts:
- Pipeline is ordered steps.
- Each step can be enabled/disabled.
- Each step has operation_type, columns, params.
- Pipeline can run in single mode or train_test mode.

Functions:
- load_pipeline(pipeline_id)
- validate_pipeline_steps(...)
- apply_pipeline_single(df, steps)
- apply_pipeline_train_test(train_df, test_df, steps)
- preview_pipeline(...)
- summarize_before_after(...)
- collect_step_effects(...)

Critical leakage-safe rule:
In train_test mode, all learned statistics must be fit on train only and then applied to test.

Examples:
- median imputation uses train median.
- scaling uses train mean/std or train min/max.
- rare category grouping learns frequent categories from train.
- one-hot encoding learns categories from train.
- clipping thresholds learned from train.
- frequency encoding learns frequencies from train.

Never fit preprocessing statistics on combined train+test.

Pipeline output:
- transformed single_df, or transformed train_df/test_df
- step_effects list
- fitted_params for reproducibility
- before_summary
- after_summary

Step effect structure:
{
  "step_id": 1,
  "operation_type": "numeric_imputation",
  "columns": ["age"],
  "summary": "Filled 42 missing values in age using train median 37.0",
  "before": {...},
  "after": {...},
  "warnings": []
}

============================================================
IMPORTANT LEAKAGE-SAFE RULE
============================================================

In train/test mode:
- Fit preprocessing statistics on train only.
- Transform train using fitted train statistics.
- Transform test using the same fitted statistics.
- Never compute imputation values, scaling statistics, clipping thresholds, category maps, or frequency maps from combined train+test.
- Generated reports and code should explicitly state this behavior.

This rule is central to the project.
